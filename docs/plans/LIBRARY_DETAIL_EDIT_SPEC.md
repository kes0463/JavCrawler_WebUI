# 라이브러리 상세 편집 명세

## 1. 품번

수정 불가(항상 읽기 전용).

## 2. 메이커

`makers` 목록·검색 → 선택 → 저장 시 `jav_metadata`의 메이커 필드에 반영.

**목록에 없음** → 「새 메이커 추가」 → 일본어·한국어·영어 입력 → `makers`에 INSERT(`ja`, `ko`, `en`, `slug` = 영어와 동일) + `jav_metadata`에도 동일 값 반영.

## 3. 배우·장르

메이커와 동일한 원칙(목록·검색·선택·새 장르/배우 추가).

**여러 개** → 필드 클릭 → 테이블 선택 UI → 칩으로 추가 → 필드 다시 클릭해 반복.

## 4. 제목·시놉시스

상단 편집 후 제목·시놉 영역 탭/버튼 → 5개 언어 팝업(일본어, 한국어, 영어, 중국어 간체, 중국어 번체).

하단 **저장** 시에만 반영, **취소** 시 이번 편집 세션 변경 폐기(팝업 확인만 하고 메인 저장 전까지는 메모리).

## 5. 출시일

캘린더(날짜 선택 UI)로 설정 가능. 저장 시 `jav_metadata`와 Grok 스토리 JSON의 `release_date`(및 캐노니컬과 짝이 되는 필드)를 같은 값으로 맞춤.

## 6. 저장 단위 — DB + Grok 함께

작품 메타를 저장할 때 `jav_metadata`와 디스크의 Grok 스토리 JSON(필요 시 `library_state.json` 포함)을 **한 번의 저장 동작** 안에서 함께 갱신한다. 출시일·제목 등 메타(1~5번)는 이 규칙에 포함된다.

## 7. 저장 원자성 및 실패 처리

- 둘 다 성공할 때만 **「저장 완료」**로 처리하는 것을 원칙으로 한다.
- 하나만 실패하면 사용자에게 알리고, **가능하면 롤백** 또는 재시도를 안내한다.

## 8. 필드 매핑 (Grok 상단 ↔ jav_metadata)

Grok 스토리 JSON 상단에는 `title_ja`, `title_ko`, `actress`, `maker`, `release_date`, `synopsis_short`, `overall_summary` 등 **제한된 키**가 있다.

영어·중국어 제목/시놉처럼 Grok 상단에 없는 언어는 `**jav_metadata`에만 두거나**, 추후 Grok 스키마를 확장할지 명시한다.

## 9. 씬 분석

### 9-1. 범위

씬 분석은 **Grok 스토리 JSON의 `scenes[]`**와 `**library_state.json`(캐노니컬)**의 씬 목록을 같은 의미로 다룬다.

씬 **추가·삭제·시간(구간) 수정**이 가능하며, 저장 시 두 저장소가 **같은 씬 집합·같은 구간·같은 텍스트 필드**를 가리키도록 한다. `scenes[]`와 캐노니컬의 씬 정보는 **동시에 일관되게** 저장한다(씬만 고칠 때도 Grok/캐노니컬이 어긋나지 않게 같은 트랜잭션에 가깝게 처리).

### 9-2. 씬 추가

새 씬에 `scene_id`(고유), `time_range` 또는 `start_sec` / `end_sec`, 라벨·요약 등 편집 가능 필드를 부여한다.

추가 저장 시 Grok JSON의 `scenes` 배열에 항목 추가 + 캐노니컬에도 동일 항목 추가.

### 9-3. 씬 삭제

목록에서 씬 삭제 시 Grok JSON과 캐노니컬 양쪽에서 같은 `scene_id` 항목 제거.

해당 씬에 묶인 스틸 경로(`still_paths`) 등이 있으면: 삭제 시 같이 비울지, 스틸 파일까지 지울지 정책을 한 줄로 정함(보통은 목록에서만 제거하고 파일은 유지하는 경우가 많음).

### 9-4. 시간(구간) 수정

시작·끝 시간 또는 `time_range` 문자열 수정 시 두 파일 모두 반영.

검증: `start_sec < end_sec`, 음수 금지, 영상 길이 초과 여부(알면) 등 기본 검사.

(씬 추가·씬 수정 공통 적용.)

### 9-5. 씬 텍스트·메타 수정

`scene_label`, `scene_summary`, `tone`, `key_tags`, `time_label` 등 Grok/캐노니컬에 있는 필드는 편집 시 양쪽 동기화.

(코드에 `locked_fields` 개념이 있으면) 사용자 확정 필드는 재생성 시 덮어쓰지 않는 규칙과 맞출지 선택.

(씬 추가·씬 수정 공통 적용.)

### 9-6. 저장 원자성

씬만 저장하는 버튼이든, 전체 저장이든 **「씬 저장」 한 번**에 대해 Grok 파일 쓰기 + 캐노니컬 쓰기를 같은 단계로 처리하고, 한쪽만 성공하면 실패로 간주하거나 사용자에게 알림.

### 9-7. 메타 편집(1~8번)과의 관계

제목·메이커 등만 고칠 때는 씬 배열을 건드리지 않아도 된다.

같은 화면에서 메타와 씬을 모두 바꾼 뒤 저장하면 **메타(DB+Grok 상단) + 씬(Grok+캐노니컬)**를 한 번의 저장으로 묶을지, 탭별 저장으로 나눌지 UX만 정하면 된다.

### 9-8. 씬 수정(기존 씬 편집) — 정의

「씬 수정」은 이미 존재하는 `scene_id`에 대해 필드를 바꾸는 모든 동작을 말한다(9-2 추가·9-3 삭제와 구분).

식별은 항상 `scene_id`로 하며, 시간만 바뀌어도 같은 씬으로 본다.

### 9-9. 씬 수정 — 구간·시간

9-4와 동일 규칙: `time_range` 또는 `start_sec` / `end_sec` 변경 시 Grok JSON + 캐노니컬 모두 갱신, 검증 동일.

### 9-10. 씬 수정 — 텍스트·태그

9-5와 동일 규칙: `scene_label`, `scene_summary`, `tone`, `key_tags`, `time_label` 수정 시 양쪽 동기화.

`key_tags`는 배열이므로 순서·중복 제거·빈 항목 제거 등 규칙을 한 가지로 고정.

### 9-11. 씬 수정 — 스틸 경로(`still_paths`)

기존 씬의 스틸 목록을 UI에서 고칠 때도 동일 `scene_id`로 Grok·캐노니컬 양쪽 `still_paths` 반영.

파일이 디스크에 없을 때는 경고만 할지 저장 불가 할지 정책 결정.

### 9-12. 씬 수정 — 표시 순서

씬 순서가 배열 순서에 의존한다면, 위/아래 이동 등으로 순서를 바꿀 때 `scenes[]` 순서를 Grok JSON과 캐노니컬 동일하게 맞춘다.

### 9-13. 씬 수정 — 잠금(`locked_fields`)

`locked_fields`를 쓰는 경우: UI에서 잠긴 필드는 읽기 전용 또는 잠금 해제 후 수정 등 동작을 정한다.

재분석·머지 시 덮어쓰기 정책과 일치시킨다.

### 9-14. 씬 수정 — 저장 실패

9-6과 동일: Grok만 반영되고 캐노니컬 실패 등 부분 성공은 사용자 알림 및 가능하면 롤백 원칙 적용.
