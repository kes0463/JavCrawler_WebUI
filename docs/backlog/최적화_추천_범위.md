# 최적화 어디까지 할지 추천

최적화는 **“지금 체감되는 병목을 얼마나 빨리/안전하게 줄일지”**에 따라 컷을 나누는 게 가장 효율적입니다. 아래 3단계 중 **2단계까지**를 기본 추천으로 두고, 3단계는 “데이터가 커져서 확실히 막혔을 때”만 진행하는 걸 권합니다.

## 1단계 (반나절~1일) — 무조건 하는 “안전한” 최적화 범위

- **불필요한 파일/캐시 정리 & git 제외**: `.pytest_cache`, `__pycache__`, DB `-wal/-shm`, 각종 `*_cache/*.json` 같은 런타임 산출물은 `.gitignore`로 확실히 제외 (성능보단 작업속도/실수방지 효과가 큼)
- **측정 먼저 넣기(가벼운 로깅)**: 처리 파이프라인(수확/스냅샷/프리뷰/하이라이트/몽타주) 각 단계에 “시작/끝/소요시간/입력 크기”만 기록
- **중복 작업 캐싱**: 동일 입력(파일 경로+mtime+설정값)인데 재생성하는 썸네일/스틸/프리뷰가 있으면 즉시 캐시 키 도입
- **UI 프리즈 제거**: QML/GUI 스레드에서 무거운 I/O/ffmpeg 호출/DB 대량 조회가 돌면 전부 워커로 이동(이미 워커가 있다면 “메인스레드에서 기다리는 코드”만 제거)

여기까지만 해도 “버벅임/재처리/불필요 반복”이 크게 줄어드는 경우가 많습니다.

## 2단계 (2~5일) — 체감 성능을 확실히 올리는 “가성비 구간” (기본 추천 컷)

- **DB 최적화**: 느린 쿼리 로그/실행계획 기준으로 인덱스 추가, N+1 조회 제거, 목록은 페이지네이션/증분 로딩
- **파이프라인 동시성 재설계(작게)**: CPU 바운드(분석) vs I/O 바운드(파일/ffmpeg) 분리해서 워커 수를 제한적으로 조절, 큐 길이/대기시간 메트릭 추가
- **이미지/비디오 처리 비용 절감**: 프리뷰는 해상도/프레임 수/샘플링 간격을 “품질 프리셋”으로 고정(High/Default/Fast), 기본은 Default
- **UI 리스트 최적화**: 포스터/썸네일 리스트는 “지연 로딩 + 스크롤 구간만 로딩(virtualization 성격)” + 이미지 디코드/리사이즈는 백그라운드

대부분의 앱은 2단계까지 하면 사용자 체감이 충분히 좋아지고, 이후는 투자 대비 이득이 급격히 줄어듭니다.

## 3단계 (1~3주+) — “규모가 커져서 반드시 필요할 때만”

- **프로파일링 기반 리팩터링**(cProfile/py-spy + flamegraph): 최상위 핫스팟만 타겟
- **네이티브 가속/바이너리화**: 병목이 Python 연산이면 numba/cython/Rust 확장 고려(단, 유지보수 비용 큼)
- **작업 재개/복구/분산 처리**: 대형 라이브러리에서 장시간 배치가 잦을 때만

## 결론 추천

- **2단계까지**를 최적화 컷으로 잡는 걸 추천합니다.
- 다만 “어디가 제일 느린지”가 아직 불명확하면 **1단계(측정+중복제거+UI 프리즈 제거)**만으로도 답이 나오는 경우가 많습니다.
